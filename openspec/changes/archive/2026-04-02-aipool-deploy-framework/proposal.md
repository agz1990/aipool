# AIPool 通用部署框架

## 概述

构建一个通用的 AIPool 部署框架，通过 Claude Code Skills 实现标准作业流程（SOP）自动化，支持 Claude Code、Gemini、Codex 等多种 AI 服务池的部署和管理。

## 问题

当前 AIPool 项目存在以下问题：

1. **部署流程依赖人工**：需要手动执行多个步骤（上传文件、SSH 登录、运行脚本、验证结果），容易出错或遗漏
2. **无状态持久化**：部署中断后无法断点续传，必须从头开始
3. **多服务器管理困难**：方案更新后，需要手动同步到每台服务器，容易出现版本漂移
4. **扩展性差**：每个新的 AI 服务池（Gemini、Codex）都需要重写一套部署流程
5. **缺乏可见性**：不知道哪些服务器部署了什么版本，健康状态如何

## 目标

### 核心目标

1. **SOP 自动化**：通过 Claude Code Skills 驱动整个部署流程，人只需在关键决策点确认
2. **状态持久化**：支持断点续传，部署中断后可以从上次位置继续
3. **多服务器管理**：自动检测版本漂移，批量同步更新
4. **Provider 抽象**：通过 Adapter 模式支持多种 AI 服务池，新增 Provider 只需编写配置文件
5. **可观测性**：清晰展示所有服务器的部署状态和健康度

### 非目标

- 不支持 Windows 服务器（仅支持 Linux 和 macOS）
- 不处理网络层面的高可用（如负载均衡、故障转移）
- 不提供 Web UI（仅通过 Claude Code CLI 操作）

## 价值

### 对用户的价值

1. **降低部署门槛**：从"需要熟悉 Linux 运维"降低到"会用 Claude Code 即可"
2. **提高可靠性**：自动化流程减少人为错误，状态持久化避免重复劳动
3. **节省时间**：多服务器批量操作，从"每台 30 分钟"降低到"一键 5 分钟"
4. **易于扩展**：新增 Gemini、Codex 池时，复用整套框架

### 对项目的价值

1. **标准化**：建立统一的部署规范，所有 AI 服务池遵循相同模式
2. **可维护性**：配置和代码分离，修改部署流程不需要改 Skill 代码
3. **可测试性**：每个步骤可独立验证，问题定位更容易

## 成功指标

1. **部署成功率**：从当前的 ~70%（人工易出错）提升到 95%+
2. **部署时间**：单台服务器从 30 分钟降低到 10 分钟（自动化部分）
3. **扩展性验证**：新增一个 Provider（如 Gemini）只需 1 天（编写 adapter.yaml + 测试）
4. **用户反馈**：至少 3 个团队成员能独立完成部署

## 范围

### 包含

1. **通用部署框架**
   - Adapter 解释器（读取 adapter.yaml 并执行）
   - 状态管理系统（inventory.yaml + state/*.json）
   - 内置操作库（mkdir、rsync、check_disk 等）

2. **Claude Code Skills**
   - `/aipool:init` - 初始化项目配置
   - `/aipool:deploy` - 部署到指定服务器
   - `/aipool:status` - 检查所有服务器状态
   - `/aipool:sync` - 同步有漂移的服务器
   - `/aipool:verify` - 验证部署健康度
   - `/aipool:rollback` - 回滚到上一版本

3. **Claude Code Pool Adapter**
   - 完整的 adapter.yaml 配置
   - 适配现有的 setup.sh 脚本

4. **文档**
   - Adapter 编写指南
   - Skills 使用手册
   - 故障排查指南

### 不包含

1. Gemini、Codex 的 Adapter 实现（留作未来扩展）
2. Web UI 或 API 接口
3. 监控告警系统（仅提供状态查询）

## 约束

1. **技术约束**
   - 必须支持免密 SSH 登录
   - 服务器需要 sudo 权限
   - 依赖 bash、rsync、jq 等基础工具

2. **安全约束**
   - 不在配置文件中存储密码或 API Key
   - 自定义脚本需要在受控环境执行
   - 状态文件不包含敏感信息

3. **兼容性约束**
   - 必须兼容现有的 claude-code-pool 部署脚本
   - 不破坏已部署的服务器环境

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| SSH 连接不稳定导致部署中断 | 高 | 状态持久化支持断点续传 |
| 自定义脚本执行失败 | 中 | 每步都有 verify 检查，失败立即停止 |
| 多服务器并发操作冲突 | 低 | 初期不支持并发，按顺序执行 |
| adapter.yaml 配置错误 | 中 | 提供 schema 验证和示例模板 |

## 替代方案

### 方案 A：使用 Ansible

**优点**：成熟的运维工具，功能强大
**缺点**：学习曲线陡峭，需要编写 playbook，与 Claude Code 集成不自然

### 方案 B：纯 Bash 脚本

**优点**：简单直接，无需额外依赖
**缺点**：无状态管理，难以扩展，缺乏抽象

### 方案 C：当前方案（Claude Code Skills + Adapter）

**优点**：
- 与 Claude Code 深度集成，用户体验最佳
- Adapter 模式提供良好的扩展性
- 状态管理支持断点续传

**缺点**：
- 需要开发新的框架
- 初期功能可能不如 Ansible 完善

**选择理由**：当前方案最符合 AIPool 项目的定位（面向开发者，使用 Claude Code），且扩展性最好。
