## Context

### 背景

中国开发团队需要通过以下架构使用 Claude Code：
1. 开发者在中国通过 ToDesk 远程到美国的 Mac 物理主机中的虚拟机
2. 在虚拟机中通过 VS Code Remote SSH 连接到开发主机
3. 在开发主机上使用 Claude Code CLI

### 当前状态

- 已有美国服务器基础设施
- 团队规模：10-15 人
- 使用模式：3-5 人同时使用，时间错开
- 预算：$100-200/月

### 约束

- 必须使用 Claude Code CLI（不是 VS Code 插件）
- 账号共享可能违反 Anthropic 服务条款
- 需要在 Linux/macOS 服务器上部署
- 团队成员需要通过 SSH 访问

## Goals / Non-Goals

**Goals:**

- 降低 Claude Code 使用成本（目标：节省 60-75%）
- 支持 3-5 人并发使用，10-15 人错峰使用
- 提供简单易用的协调机制，避免冲突
- 10 分钟内完成部署
- 提供完整的监控和管理工具
- 编写详细的用户和管理员文档

**Non-Goals:**

- 不支持 VS Code 插件方式的共享
- 不保证 100% 合规（有一定风险）
- 不支持多地理位置同时使用
- 不提供 Web 界面（v1.0）
- 不支持细粒度的权限控制

## Decisions

### 决策 1: 使用共享 Linux 账号而非容器化

**选择**: 为每个池子创建独立的 Linux 用户账号

**理由**:
- 简单直接，无需额外的容器技术
- 用户可以直接 SSH 切换账号
- 资源隔离足够（不同用户账号）
- 部署和维护成本低

**备选方案**:
- Docker 容器：增加复杂度，需要管理容器网络和存储
- 虚拟机：资源开销大，不适合这个场景

### 决策 2: 使用文件锁而非数据库

**选择**: 使用简单的文件锁（.lock 文件）来管理池子状态

**理由**:
- 无需额外的数据库依赖
- 文件系统原子操作足够可靠
- 易于调试（直接查看文件）
- 符合 Unix 哲学

**备选方案**:
- Redis/SQLite：增加依赖，过度设计
- 进程锁：不够持久，进程退出后丢失

### 决策 3: 使用 Bash 脚本而非编程语言

**选择**: 所有工具使用 Bash 脚本实现

**理由**:
- 服务器环境自带 Bash
- 无需安装额外运行时（Python/Node.js）
- 易于修改和定制
- 符合系统管理工具的传统

**备选方案**:
- Python：需要安装，但提供更好的错误处理
- Go：需要编译，部署复杂

### 决策 4: 共享代码目录使用软链接

**选择**: 在 `/shared/repos` 创建共享目录，每个池子账号通过软链接访问

**理由**:
- 避免代码重复
- 便于团队协作（同一份代码）
- 节省磁盘空间
- 符合 Unix 文件系统惯例

**备选方案**:
- NFS 挂载：过度复杂
- 每个池子独立目录：浪费空间，不便协作

### 决策 5: 分阶段部署策略

**选择**: 先部署 2 个池子测试，再扩展到 4 个池子

**理由**:
- 降低初期风险
- 验证技术可行性
- 收集真实使用数据
- 根据反馈调整

**备选方案**:
- 一次性部署 4 个池子：风险高，浪费资源（如果不够用）

## Risks / Trade-offs

### 风险 1: 合规性风险

**风险**: 账号共享可能违反 Anthropic 服务条款，导致账号被封禁

**缓解措施**:
- 先小规模测试（2 个池子）
- 监控是否收到警告
- 准备备选方案（Teams 计划或 API 方案）
- 使用量保持在合理范围
- 避免 24/7 连续使用

### 风险 2: 并发冲突

**风险**: 高峰期所有池子都被占用，用户需要等待

**缓解措施**:
- 根据实际使用数据调整池子数量
- 建立使用规范（使用完立即释放）
- 提供实时状态查看
- 考虑混合方案（核心开发者独立订阅）

### 风险 3: 数据安全

**风险**: 共享目录中的代码对所有池子可见

**缓解措施**:
- 明确告知团队成员
- 敏感项目不放在共享目录
- 定期清理敏感数据
- 使用 Git 管理代码（不在本地存储敏感信息）

### 风险 4: 单点故障

**风险**: 开发主机故障导致所有人无法使用

**缓解措施**:
- 定期备份共享代码
- 准备备用服务器方案
- 监控服务器健康状态

### Trade-off 1: 成本 vs 合规性

**选择**: 优先降低成本，接受一定的合规风险

**理由**:
- 团队预算有限
- 可以先测试，如果有问题再转向合规方案
- 备选方案已准备好

### Trade-off 2: 简单性 vs 功能完整性

**选择**: v1.0 保持简单，不提供 Web 界面和高级功能

**理由**:
- 快速上线，验证可行性
- 命令行工具对开发者足够友好
- 后续可以根据需求迭代

## Migration Plan

### 部署步骤

1. **准备阶段**（1 小时）
   - 上传部署脚本到开发主机
   - 检查系统要求（磁盘空间、权限）
   - 准备 Claude Code 订阅账号

2. **部署阶段**（10 分钟）
   - 运行部署脚本 `setup.sh -n 2`
   - 为每个池子登录 Claude Code
   - 验证部署成功

3. **测试阶段**（1-2 周）
   - 让 2-3 个人试用
   - 收集反馈和使用数据
   - 调整配置

4. **扩展阶段**（10 分钟）
   - 根据测试结果扩展到 4 个池子
   - 全员培训
   - 正式上线

### 回滚策略

如果遇到问题（如收到 Anthropic 警告）：

1. **立即停止使用**
   - 强制释放所有池子
   - 通知团队停止使用

2. **评估备选方案**
   - Claude Teams 计划
   - 每人独立订阅
   - Claude API 自建服务

3. **快速切换**（24 小时内）
   - 根据选择的方案执行切换
   - 更新文档和培训

### 监控指标

- 池子利用率（目标：60-80%）
- 冲突频率（目标：< 2 次/周/池）
- 用户满意度（目标：> 80%）
- 成本节省（目标：> 60%）
